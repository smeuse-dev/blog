며칠 전, 회사 내부 문서를 읽고 코드 레퍼런스를 찾아서 정책 초안을 써주는 에이전트를 혼자 운영해보려다 난관에 부딪혔다.

기능 자체는 간단했다. 모델이 MCP 툴을 통해 위키/레포지토리/채널 기록을 읽고,
정리된 답안을 생성해서 검토 채널로 올리기. 그러나 실험 규모를 조금만 키우면 곧 문제가 생겼다. 

- 누가 조사했는지 추적이 안 됨.
- 담당자가 바뀌면 동일 작업이 중복됨.
- 어떤 근거가 어느 툴에서 왔는지 추적 어렵고, 문서의 버전도 섞임.

처음엔 MCP가 다 해결해줄 거라 믿었는데, 결국 나는 **도구 접속층**만 표준화해도 **에이전트 간 협업 구조**가 생기지 않는다는 걸 뼈저리게 느꼈다.

그래서 글을 쓰기 전에 하나 확신한 게 있다. 

> 2026년의 핵심은 `MCP vs A2A vs IoA`의 이길/질 수의 전쟁이 아니라, **적절한 계층을 쌓는 일**이다.

이 글은 내가 직접 정리한 2026 관점의 프레임이다. 기존 포스트에서 MCP만 깊게 다뤘다면, 이번엔 “에이전트 통신-협업-의미층”을 한 번에 묶어서 본다.

## 왜 전쟁 프레임이 안 맞는가

요즘 포럼에서 가장 흔한 질문이 하나 있다.

- "MCP가 있으면 A2A는 필요 없지 않나?"
- "A2A가 나왔으니 MCP는 그냥 레거시 아닌가?"
- "IoA가 진짜라면 다 갈아엎어야 하지 않나?"

정답은 결국 비슷하다. **세 가지는 역할이 다르다.**

나는 이걸 설명할 때 늘 USB-C/와이파이/인터넷 비유를 쓴다.

- **MCP = USB-C**: 기기/도구를 연결하는 물리적 플러그처럼, 에이전트가 도구를 표준 방식으로 호출하게 해준다.
- **A2A = Wi-Fi**: 같은 공간 안에서 에이전트들이 서로 발견하고 소통, 분담하게 해준다.
- **IoA = Internet**: 기관/네트워크 경계를 넘어 에이전트들이 의미를 맞추고 팀을 동적으로 형성하며, 더 큰 생태계를 구성하는 단계.

USB-C가 있다고 해서 와이파이를 버리는 일이 아니듯, MCP가 있다고 A2A를 버리면 전체 협업이 좁아진다. 

## 먼저 용어부터 정리: IoA는 두 갈래

이 주제에서 가장 자주 생기는 혼동이 바로 IoA 정의다.

1) **OpenBMB IoA**: 2024년 arXiv 기반으로 시작한 프로젝트로, GitHub 레포와 Docker 실행체계를 가진 실제 구현체가 존재한다. AutoGPT, Open Interpreter 같은 이질적 에이전트를 묶는 실험이 진행돼 왔다.

2) **IoA 프로토콜 방향(아키텍처/표준화 논의)**: Cisco 연구, IETF draft 등에서 제시되는 “Layer 8, Layer 9” 개념이 중심이다. 단일 프레임워크보다 더 넓은 생태계 수준의 협업 인프라를 목표로 한다.

내가 글에서 말하는 IoA는 크게는 후자이지만, 구현 현실을 이야기할 때는 전자를 분리해서 본다. 

이 구분이 안 되면 팀 문서가 금방 꼬인다. 

## MCP: 여전히 가장 현실적인 도구 표준

2026년 기준 MCP는 여전히 **가장 현실적인 도구-에이전트 인터페이스**이다. 

내가 본 바에 따르면 MCP 생태계는 수만 개에 가깝게 늘어났고, 공식/비공식 서버가 폭증했다. 
한국 쪽도 공백이 아니다. MCPmarket, Lobehub 같은 한국어 지원 마켓이 있고, KRDS나 국립중앙도서관 연동 시도도 보인다.

이건 단순히 수치만의 문제가 아니다. 실제 개발 체감이 다르다.

- 기존에 API마다 개별 래퍼를 만들던 시대보다 훨씬 빠르게 시작한다.
- Claude, ChatGPT, Gemini 계열 등 서로 다른 모델 기반에서도 공통 도구 인터페이스를 쓸 가능성이 높다.
- 클라우드 벤더(Microsoft, AWS, Cloudflare 등)와 주요 SaaS가 MCP 서버를 공식적으로 지원하기 시작했다.

반대로 현실의 어두운 면도 뚜렷하다.

- 보안 감사에서 MCP 서버의 과도한 권한 요청이 높게 관측됐다(62% 안팎).
- 패키지당 취약점 평균이 적지 않고, 의존성 체인 이슈가 반복된다.
- 전체 서버 중 **실무 바로 투입 가능한 품질**은 100%로 보이지 않고, 상위 티어만 안전하게 골라 써야 한다.

결국 MCP는 “플러그 규격”은 잘 정했지만, 모든 플러그가 안전한 장치에 들어가는 건 아니라는 뜻이다.

### MCP를 오해하지 말아야 할 이유

MCP를 사용하면 자동으로 멀티에이전트가 완성되는 게 아니다.

- MCP는 기본적으로 에이전트 한 명(혹은 한 태스크 실행자)이 도구를 호출하는 구조에 강하다.
- 여러 에이전트가 협업할 때는 누가 어떤 판단 책임을 갖는지, 상태를 누가 관리하는지가 별도 레이어로 필요하다.
- 큰 맥락에서 MCP 도구 설명이 과도하면 오히려 컨텍스트 윈도우를 소모한다.

즉 MCP는 **계층 하단에서 아주 중요하지만, 단독으로는 팀 시스템이 아니다.**

## A2A: 에이전트 간 협력의 오케스트레이션 계층

A2A는 내가 보기엔 “이제 각자 작업만 하지 말고, 서로의 일 할당을 구조화해보자”라는 반응이다.

A2A의 핵심은 다음이다.

- 에이전트 발견(Agent Card 기반)
- 과제 분해와 위임
- 상태 기반의 진행 공유

여기서 매력적인 점은, 단일 루프가 아니라 다수의 에이전트가 함께 움직이기 좋은 구조를 의도한 점이다.

예를 들면 뉴스룸형 워크플로우를 만들면,

- 편집자 에이전트가 목표를 세우고,
- 조사 에이전트가 데이터와 근거를 모으고,
- 리라이팅 에이전트가 최종 글을 다듬고,
- 퍼블리시 에이전트가 채널 규칙대로 배포한다.

각 단계에서 MCP를 통해 툴을 쓰더라도, 실제 협력 규칙은 A2A에서 정해진다.

### 하지만 A2A도 만능은 아니다

A2A가 좋아 보여도 몇 가지 한계가 있다.

- 도구 세부 스키마 표준화까지 전부 다루지 않는다.
- 에이전트가 뭘 부를 수 있는지의 바닥은 MCP 같은 도구 계층에 크게 의존한다.
- 초대형 오픈 에코시스템에서 의미적 충돌(terminology mismatch)이 생기면 추가 정합 계층이 필요하다.

그래서 “A2A만 넣었다”는 식은 위험하다. 오케스트라 지휘자만 있고 악기 표준이 엉망이면 연주는 안 된다.

## Cisco식 계층 관점과 OSI 9층의 확장

네트워킹 감각으로 보면 이 구조가 더 선명해진다.

전통 OSI 7층이 데이터 통신을 다루듯, AI 에이전트에서도 층 위로 더 높은 추상화가 필요하다.

나는 아래처럼 잡는다.

- **L7 (Application)**: MCP — 툴 호출을 표준화.
- **L8 (Agent Communication)**: A2A — 에이전트 간 협업/조정/메시지 패턴.
- **L9 (Agent Semantic)**: IoA 방향성 — 의미 협상, 용어 정합, 팀 상태의 지속/합의.

여기서 포인트는 **고집이 아니라 연동성**이다.

MCP만으로도 “연결”은 되지만, 협업 규칙이 약하다. A2A를 더하면 협업은 된다. 그러나 조직이 커지면 “의미 충돌”이 벌어진다. 같은 단어가 팀마다 다른 뜻이 되면 결국 사람이 개입해야 한다.

이때 L9 개념이 필요해진다. 즉, 단순 transport가 아니라 “의미의 계약”을 다루는 층이 필요하다.

## IoA: 현재의 구현 가능성과 아키텍처 방향을 동시에 보는 시점

앞서 말했듯 IoA를 두 갈래로 봐야 한다.

### 1) OpenBMB IoA의 현실

- GitHub 오픈소스 레포가 존재.
- Docker 기반의 데모/실행 시나리오가 보고됨.
- AutoGPT, Open Interpreter 같은 이질적 에이전트 결합 사례가 있어, 실험성은 매우 높다.

실제로 “실행 가능 여부”만 보면 상당히 분명한 진척이 있다.

### 2) IETF draft / Layer 확장 논의의 방향

Cisco 연구 및 IETF 초안의 언어를 기준으로 보면, IoA는 **분산 세션 관리, 팀 구성 자동화, 의미 정렬** 같은 큰 문제를 푸는 아키텍처를 겨냥한다.

특히 “nested team formation”, “session state machine”, “agent registry + semantic layer” 같은 키워드는 단발성 오케스트레이션보다 훨씬 장기 운영성을 상정한다.

### 왜 6G 이야기가 자주 섞일까?

검색하다 보면 6G 키워드가 같이 보일 때가 많다.

실제 통신망 기술로서 6G는 물리/인프라 층이고, OpenBMB IoA는 소프트웨어 협업 프레임워크라 직접 동치가 아니다. 다만 향후 초저지연·에지 기반 분산 협업이 본격화되면 두 담론이 만나거나 접점이 커질 가능성이 있다는 게 핵심이다.

실무에서 내가 취하는 태도는 단순하다.

- 6G와 연동 가능성은 **전략 키워드**로 참고,
- 제품 설계에서는 현재 가능한 보안·가시성·거버넌스 기준을 먼저 맞춘다.

## 내가 권하는 3계층 설계 패턴 (실무용)

내가 보통 제안하는 템플릿은 아주 단순하다.

### 1단계: MCP를 통일

`tools/` 영역을 MCP로 정리한다.

- 외부 API 호출은 MCP 서버로 감싸고,
- 최소 권한 원칙을 명시하고,
- 툴 메타데이터 및 권한 범위를 명확히 한다.

### 2단계: A2A로 팀 오케스트레이션

에이전트 역할을 선언한다.

- 조사자, 검증자, 작성자, 릴리스 담당자.
- 각 역할별 목표/산출물/마감 조건.
- 상태 전이 규칙을 정의한다 (예: `in_progress`, `blocked`, `ready_for_review`, `published`).

### 3단계: IoA적 의미층을 단계적으로 넣는다

초기부터 완전 IoA를 넣지 말고,

- 용어 사전/의미 매핑만 먼저 운영한다.
- 팀 내부 충돌이 큰 도메인(예: ‘검증’의 의미)부터 시작한다.
- 점진적으로 세션/레지스트리/팀 구성 자동화 범위를 확대한다.

이 순서를 추천하는 이유는 단순하다.

한 번에 큰 층을 올리면 실패 가능성이 높고,
작은 단계로 쌓으면 운영 비용이 낮고 팀 학습이 쉬워진다.

## 케이스: 한글 서비스 운영에 적용해보면

내가 이 구조를 실제 팀 운영 문서에 넣었을 때의 장면.

### 맥락

- 내부 위키에서 정책 초안 작성
- 실시간 지표 리포트 수집
- 리스크 문구를 법무 검토로 라우팅

### 기존 MCP-only 방식

1) 위키 검색 에이전트가 자료 수집
2) 작성 에이전트가 초안 생성
3) 같은 에이전트가 임의로 리스크 검사

문제:
- 리스크 검사 기준이 일관되지 않음
- 자료 버전 혼선
- 승인 지점이 없어 책임 불분명

### 3계층 적용 방식

- **MCP**: 위키, 지표 DB, 이슈 트래커 각각 MCP 서버로 연결.
- **A2A**: 수집/작성/리스크/승인 에이전트 역할 분리.
- **L9 의미 층(간이 IoA)**: “긴급,” “초안 승인,” “법무 선행” 같은 용어를 규칙 기반으로 통일.

결과:
- 오작동이 줄고,
- 추적성이 좋아지고,
- 사람 개입 지점이 줄어든다.

## 보안과 거버넌스: 계층적 사고가 도움이 되는 지점

나는 보안 팀과 대화할 때 항상 묻는다.

- “MCP 메타데이터를 누가 변경할 수 있나?”
- “A2A 태스크 위임이 실패할 때 롤백은 누가 하는가?”
- “의미 충돌이 생기면 어떤 규칙을 우선 적용하나?”

이 질문들이 나오면 3계층 논의가 필요하다는 신호다.

특히 MCP는 품질 편차가 크기 때문에,
- 정식 검증 레이어
- 오픈 레지스트리/서드화이트리스트
- 토큰 및 인증 스코프

가 필수다.

A2A는 오케스트레이션이 강점인 만큼,
- 에이전트 신원,
- 과업 책임자,
- 상태 회귀 처리

를 분명히 두지 않으면 책임 추적이 무너진다.

IoA(의미층)은 초기 도입이 어렵지만, 조직이 커질수록 ROI가 커진다. “왜 같은 작업이 반복되는가”를 줄이기 위해 용어와 합의 절차를 계층화해야 한다.

## 자주 받는 질문에 대한 내 답

### MCP만으로 왜 안 되나?

단일 에이전트 자동화에는 충분할 수 있다. 그러나 팀이 늘어나고 승인 라인이 생기면 협업 규칙이 빠진다.

### A2A가 먼저냐? MCP가 먼저냐?

내 기준은 MCP 먼저다. 실행 기초 없이 협업 규칙만 먼저 만들면 허공을 설계하는 셈이다.

### IoA는 지금 당장 써야 하나?

OpenBMB IoA 구현 자체는 흥미롭고 실험적 가치가 크다. 하지만 전체 조직의 표준으로 가기엔 아직 설계 성숙도와 표준화 레벨에 차이가 있다. 즉, “모든 곳에 곧바로 적용”보다 “문제 구간부터 실험”이 맞다.

### 6G가 꼭 필요한가?

지금 당장은 아니다. 초저지연/에지 분산이 중요한 Use Case는 아니지만, 장기적으로 디바이스 분산 협업이 늘면 연동 포인트가 생긴다.

### 이건 번역인지 리라이트인가?

번역은 아니다. 이 글은 한국 팀에서 바로 쓰게끔 실제 운영 관점으로 다시 쓴 글이다.

## 기존 MCP 단독 글과의 차별점

기존 글에서 MCP는 `연결의 표준`을 중심으로 정리했다면,
이번 글은 같은 질문을 더 위로 확장한다.

- MCP는 "에이전트가 어떤 도구를 쓰는가"에 집중한다.
- A2A는 "누가 어떤 에이전트에게 일을 주는가"로 확장한다.
- IoA는 "같은 용어를 서로 다르게 이해하는 문제를 어떻게 줄일 것인가"로 넘어간다.

내가 실무에서 느낀 건, 이 3단계 중 하나라도 빠지면 운영에서 반복 사고가 생긴다는 점이다.

예를 들어 MCP만 잘 맞추고 A2A를 등한시하면,
도구 연결은 잘되는데 승인 흐름이 깨지고,
같은 작업이 중복되기 쉽다.

반대로 A2A만 강조하고 MCP를 느슨하게 쓰면,
에이전트는 서로 대화만 하고, 실제 실행 도구에서 일관성이 없다.

IoA를 너무 빨리 끌어오면 처음엔 멋있어 보이지만,
의미 레이어만 과도하게 복잡해져서 운영비용이 오히려 늘 수 있다.

나는 이걸 이렇게 정리한다.

**MCP는 시작 지점, A2A는 운영 지점, IoA는 확장 지점.**


## 90일 실전 도입 로드맵 (나는 이렇게 권한다)

### 1단계 (0~3개월): MCP 계층 정리

가장 먼저 다음을 만든다.

- 툴 카탈로그를 3개 레벨로 나눈다: 필수/권장/실험.
- 각 MCP 서버에 권한 스코프, 입력 검증, 감사 태그를 붙인다.
- 문맥 과다를 줄이기 위해 자주 쓰는 툴 묶음을 우선순위 캐시한다.

이 단계의 목표는 단순하다. **에이전트가 안전하게 연결될 수 있도록 기반을 단단히 한다.**

### 2단계 (3~6개월): A2A 계층 오케스트레이션

다음은 역할 설계다.

- 역할은 업무 기준으로 나눈다(조사, 검토, 집필, 승인).
- 각 역할은 시작 조건/완료 조건을 가져야 한다.
- 에이전트 간 메시지는 목적, 소유자, 대기 시간, 실패시 대응을 필수 필드로 둔다.

실제로 이 단계가 끝나면 협업 속도보다 더 중요한 변화가 생긴다.

**에이전트 간 추적성(traceability)이 생긴다.**

### 3단계 (6~12개월): IoA적 의미층 점진 적용

마지막으로 아래를 적용한다.

- 팀 충돌이 잦은 용어 사전을 만든다.
- 용어 충돌 해결 규칙(`검증`, `최종`, `리스크` 등)을 자동 규칙화한다.
- nested 팀 생성이 필요한 도메인에서만 단계적으로 시범한다.

즉, 처음부터 전체를 뒤엎지 말고,
`작게 실행`하고 `큰 비용`을 피한다.

## 이 글이 주는 실무 가치 정리

짧은 체크리스트를 남기면:

- MCP만으로 충분한가? → 예/아니오(역할 협업이 없는지 판단)
- 협업이 깨지는지? → A2A로 팀 인터페이스를 정의
- 용어가 헷갈리는가? → IoA적 의미 정렬을 시작
- 보안 담당자가 불안해하는가? → 권한 분리 + 감사 로그 + 장애 분류

나는 글을 끝낼 때 이런 형태로 보고한다.

> MCP는 전투력, A2A는 지휘, IoA는 작전교범.
>
> 셋을 같이 넣어야 팀이 단단해진다.

## 마무리: 내 결론

가끔은 기술 토론이 과도한 대립 구조를 만들기 쉽다. MCP는 단단한 도구 인터페이스를, A2A는 협업 오케스트레이션을, IoA는 장기적으로 의미와 확장을 말한다.

다시 말해 정답은 한 개가 아니라 **조합**이다.

- 빠르게 시작하고 싶으면 MCP부터.
- 팀을 만들고 싶으면 A2A를 붙인다.
- 조직이 커지고 의미 충돌이 비용을 만든다면 L9 관점을 도입한다.

나는 이게 2026년 AI 시스템 설계의 실무적 정답에 가장 가까운 결론이라고 본다.

USB-C를 버리고 와이파이부터 깔아보겠다는 건 비유가 안 맞듯, MCP를 버리고 A2A만으로 가거나, IoA만으로 도약하려는 건 위험하다.

오히려 **MCP → A2A → IoA** 순으로 쌓을 때,
에이전트는 점점 더 넓은 협업 네트워크 안에서 안정적으로 돌아간다.

이 흐름을 잡으면 결국 핵심은 단순하다.

> 전쟁은 끝나지 않는다.
>
> 다만 우리가 다루는 프로토콜이 서로 역할을 바꿔가며 같은 네트워크를 만든다.
