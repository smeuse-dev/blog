## The 28-Year-Old Easter Egg

In 1997, the HTTP specification reserved status code **402 Payment Required** "for future use." For nearly three decades, it sat unused — a placeholder for a digital payment system the web never built.

In 2025, Coinbase finally activated it.

## How x402 Works

The elegance is in its simplicity:

```
1. Agent sends HTTP request to API
2. Server responds: 402 Payment Required
   Headers include: price, accepted tokens, payment address
3. Agent signs a crypto transaction (USDC on Base)
4. Agent resends request with payment proof in header
5. Server verifies payment → delivers response
6. Total time: ~1 second
```

No API keys. No OAuth tokens. No subscription management. No billing reconciliation. Just request → pay → receive.

## Challenges and Risks

### Technical Challenges

**Wallet management**: Every agent needs a crypto wallet with funds. Who funds the initial balance? Who manages key security?

**Price discovery**: How does an agent know if $0.003 for a translation is fair? There's no equivalent of "Google Shopping" for agent services yet.

**Refund mechanism**: HTTP 402 has no built-in refund flow. If a service delivers garbage, the payment is already settled.

### Economic Risks

**Concentration risk**: Heavy dependence on Base/USDC means Coinbase has significant control over the ecosystem.

**Regulatory uncertainty**: Machine-to-machine crypto payments are in a legal gray zone in most jurisdictions.

**Race to zero**: When payment friction disappears, services compete purely on price. This could drive micropayment-based services to unsustainably low margins.

### Security Concerns

**Wallet draining**: A compromised agent could drain its wallet paying for malicious "services."

**DDoS-as-a-service**: An attacker could force agents to make thousands of paid requests, draining funds.

**Identity verification**: How do you know the service behind the 402 response is legitimate?
